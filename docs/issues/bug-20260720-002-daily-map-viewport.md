# BUG-20260720-002：现场切换每日行程后地图仍停留在前一天

- 状态：In Progress / G2（PR #74 已合入，等待受控环境与真机复验）
- 优先级：P1
- 父任务：[`TASK-APP-001`](task-app-001-android-basic.md)
- 关联 Requirement：`REQ-UX-003`、`REQ-APP-001`
- 最近更新：2026-07-20

## 目标

现场执行页切换到其他天时，让“本天”地图同步适配所选日期的有效地点，避免日期内容已更新而地图仍停留在前一天。

## 范围内

- 将日期切换作为显式地图视口意图，重新适配所选日期的有效地点与路线。
- 保持后台坐标刷新、打卡状态刷新时不抢夺用户已经手动调整的视野。
- 增加确定性高德地图 Mock 浏览器回归，覆盖等量地点的跨天切换。

## 范围外

- 不修改全局行程地图的展示范围、路线模式或地图 SDK 配置。
- 不修改地点坐标、后端 API、持久化数据或手机 GPS 定位。
- 不调用真实高德地图、AI 或其他外部服务。

## 复现与根因

- 确定性浏览器用例先显示第 1 天北京地点，模拟用户手动调整地图后切换到仅含 1 个成都地点的第 2 天；日期按钮已切换，但 3 秒内没有新的 `setFitView` 调用，用例按预期失败。
- `CheckinMap` 仅在首次加载、路线模式变化或可见覆盖物数量增长时自动适配视口。两天有效地点数量相同，覆盖物数量保持为 `1`；日期切换也没有独立的视口作用域信号。
- 用户手动操作后 `userAdjustedViewportRef` 会继续阻止普通数据更新抢夺视野；日期切换属于新的显式用户意图，应只在该作用域变化时重置保护并强制适配。

## 验收标准

- [x] “本天”地图初始显示第 1 天后，切换第 2 天会重新适配第 2 天的有效地点。
- [x] 即使两天有效地点数量相同，地图也不会因覆盖物数量未增长而停留在前一天。
- [x] 用户手动调整过视野后，切换日期仍以新日期为显式意图重新定位。
- [x] 普通后台数据刷新继续保留现有视口保护，不引入地图实例重建。
- [x] 地图定向 Playwright、Web lint、typecheck 与生产构建通过。

## 验证

- 复现与回归：`pnpm test:e2e map-route-regression.spec.ts`
- 静态检查：`pnpm lint`、`pnpm typecheck`
- 构建：`pnpm build`
- Eval：不适用；本缺陷不涉及 AI 输出或模型行为。

## 验证结果

- 新增用例修复前按预期失败：日期已切到第 2 天，但 3 秒内没有新的 `setFitView` 调用；修复后同一用例通过。
- 地图路线、标记生命周期和刷新初态三组 Playwright 回归 15/15 通过，覆盖后台刷新不重建地图与延迟坐标/轨迹重新适配。
- `pnpm lint`、`pnpm typecheck`、`pnpm docs:check`、`pnpm work-items:check` 与 `git diff --check` 通过。
- `pnpm test` 通过，Server 681/681；`pnpm build` 通过。首次构建清理 E2E 遗留 `.next` 符号链接时遇到 Windows `EPERM`，隔离并清理可再生成产物后重试成功。
- 浏览器回归使用确定性高德地图 Mock；未调用真实地图、AI、付费服务或真实用户数据。
- PR #74 已受检合入产品基线 `ae0700a`；本地 rebase 后的浏览器复跑曾受 Playwright revision 下载环境阻塞，不将浏览器安装失败记为应用失败。
