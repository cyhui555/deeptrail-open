# BUG-20260717-001 服务器与本地任务耗时差异分析

- 状态：分析完成；修复已发布并完成目标环境 G3
- 关联 Work Item：[BUG-20260717-001](../issues/bug-20260717-001-optimize-watchdog-race.md)
- 发现版本：目标环境 `v0.2.0`
- 分析日期：2026-07-17

## 结论

服务器与本地耗时差距主要来自比较口径、真实外部依赖和部署版本不同，现有证据不支持 CPU 或内存不足是首要根因：

- 本地自动化使用确定性 AI/地图替身；服务器运行真实 LongCat 与 Nominatim，不能直接比较。
- 问题任务生成 7 天、`39,112` tokens；本地受控探针仅 1 天、`7,525` tokens，输出量约相差 `5.20` 倍。
- 服务器约 48 个 POI 串行地理编码；连接失败时，超时、一次重试和退避累计可接近 9 分钟。
- 生产 `durationMs` 覆盖 AI、解析、坐标/交通补全及部分持久化，不等于模型推理时间。
- 修复已随不可变 Release 发布；目标机断网固定回归验证熔断、取消传播与终态竞争边界。

## 脱敏时间线与口径

| 时间/指标 | 事实 |
| --- | --- |
| 17:46:09 | `OPTIMIZE` 开始；Nominatim 随后持续连接超时 |
| 17:56:30 | 10 分钟 watchdog 写入 `FAILED`，旧工作线程未停止 |
| 18:04:11 | 线程结束：`durationMs=1081521`、`tokens=39112`、`dayCount=7`，迟到结果被丢弃 |
| `AiChatGateway.latencyMs` | ChatClient 调用到取得响应，接近 Provider 延迟 |
| `AiCallSummary.durationMs` | AI 编排到增强和成功审计准备结束，包含后处理 |

修复前单 POI 故障路径上界约为 `5 秒 × 2 + 1.2 秒 = 11.2 秒`；48 项约 `537.6 秒`。这是按配置推导的边界，不是逐条日志实测和，但足以解释 10 分钟预算被坐标补全消耗。

## 已验证与未知项

- 目标入口公开探针约为 API `34.7 ms`、Web `28.9 ms`，Compose 无显式 CPU/内存硬限制；未观察到基础入口普遍阻塞。
- 未取得目标机 CPU/load、内存/swap、容器 OOM/重启、到 LongCat/Nominatim 的分阶段网络耗时，故资源争用仍属未证实项。
- `main@6495d3e` 已统一 deadline、传播取消、竞争唯一终态、仅在提交完成后写成功审计，并对 Provider 完整失败短时熔断。
- 本地 Server `655/655`、Server E2E `37/37`、Smoke `11/11`；这些不能替代目标环境 G3。

## G3 验证

1. 以不可变 Release 发布包含 `6495d3e` 的主干版本。
2. 用脱敏 48 POI 故障夹具证明 deadline、熔断、无迟到 Provider 调用和无错误成功审计。
3. 使用同一任务类型、天数、Provider 和开关分别对比“真实 AI/关闭地理编码”“固定 AI/真实地理编码”“固定 AI/故障地理编码”。
4. 只记录 `queueMs/providerMs/parseMs/geocodeMs/transportMs/persistenceMs/totalMs` 等聚合指标，不保存用户内容。

最终判定：主要原因是不可直接比较的真实链路，次要原因是串行地理编码超时累计；大模型输出量是重要因素，服务器硬件瓶颈未证实，目标环境 G3 仍为 `pending`。
